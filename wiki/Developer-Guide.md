# Developer Guide

This page is for developers maintaining the FSW Booking System code. It covers the
repository + `clasp` workflow, the module breakdown, the project's core conventions,
the request/routing model, and a recipe for adding a new client-callable feature.

For the architectural overview see [[Architecture]]; for the per-tab schema see
[[Data Model]]; for the exact server function signatures see [[API Reference]]; and
for the unit-test suite see [[Testing]].

## Repository layout

```
FSW-Booking-System/
  package.json        # clasp scripts + devDependencies
  .claspignore        # only src/ is pushed to Apps Script
  .clasp.json         # scriptId (generated by `clasp create`)
  README.md
  docs/               # per-phase docs (.md sources + generated .docx)
  tools/              # md2docx.py + build-docs.sh (NOT pushed)
  src/
    appsscript.json   # manifest: timezone, V8, webapp, scopes
    Schema.gs  Config.gs  TimeUtil.gs  SheetDAL.gs  AuditLog.gs
    AvailabilityEngine.gs  Setup.gs  Tests.gs
    Code.gs  Auth.gs  AvailabilityApi.gs  BookingApi.gs  TimeOffApi.gs
    Notifications.gs  Reminders.gs  Triggers.gs  Reporting.gs
    ui/  Index.html Styles.html JsCommon.html Notice.html
         Employee.html JsEmployee.html  Manager.html JsManager.html
```

Only `src/` ships to Apps Script. Everything else (docs, tooling, plans, git
metadata) stays in the repo.

## clasp workflow

`clasp` is the Google command-line tool that syncs the `src/` folder with the
Apps Script project. It is a `devDependency` (`@google/clasp`) and is driven via
the `package.json` scripts.

| Task | Command (`npm run …`) | Underlying clasp |
|---|---|---|
| Sign in (one-time, as the owner) | `npm run login` | `clasp login` |
| Create the script project | `npm run create` | `clasp create --type webapp --title "FSW Booking" --rootDir src` |
| Push code | `npm run push` | `clasp push` |
| Push + auto-resync on save | `npm run pushw` | `clasp push --watch` |
| Open the editor | `npm run open` | `clasp open` |
| Create a deployment | `npm run deploy` | `clasp deploy` |
| List deployments | `npm run deployments` | `clasp deployments` |
| Tail execution logs | `npm run logs` | `clasp logs` |

If you are taking over an existing script, run
`clasp clone <scriptId> --rootDir src` instead of `create`, then `clasp push`.

### What gets pushed: `.claspignore`

`.claspignore` keeps everything except `src/` out of `clasp push`. It excludes
`node_modules`, `.git`, `package.json` / lock files, all `*.md`, `.gitignore`,
`.claspignore` itself, `.clasp.json.example`, and `plans/`. Keep non-runtime
files out of `src/` so they are never uploaded to the Apps Script project.

### Dev vs prod

Apps Script distinguishes two URLs per deployment:

- **`/dev`** — the editor's "Test deployments" URL. Always serves the **latest
  saved code** (what you just pushed). Use it while developing and for the
  verification checks below.
- **`/exec`** — a **published** deployment URL. Serves a pinned version. The
  Google Sites embed must point at `/exec`.

Re-deploying to the **same deployment id** keeps the `/exec` URL stable, so the
Site embed never breaks. Update in place with
`clasp deploy --deploymentId <id>` (find ids via `clasp deployments`). Full
deployment and embedding details are in [[Deployment]].

## Module breakdown and responsibilities

The codebase is layered: a business-rule-free foundation, an identity layer, the
client-callable APIs, and the HTML/JS UI.

| Module | Responsibility |
|---|---|
| `Schema.gs` | Single source of truth for the data model: tab names (`SHEETS`), column order (`HEADERS`), primary keys (`PK`), column formats, and all enums (`ROLES`, `FREQ`, `BOOKING_STATUS`, …). |
| `Config.gs` | Reads/writes the `Config` tab; provides `getConfig` / `getTz` and the `DEFAULTS`. |
| `TimeUtil.gs` | All time math: epoch-ms ⟷ wall-clock conversions in the business timezone, formatting (`formatHuman`), interval helpers. |
| `SheetDAL.gs` | The single choke point for all Sheet I/O: `readObjects`, `appendObject`, `updateById`, `findById`, caching, and `withScriptLock`. |
| `AuditLog.gs` | Appends immutable audit entries (who/what/when, before/after JSON) to the `AuditLog` tab. |
| `AvailabilityEngine.gs` | Pure availability computation: expands recurring rules, subtracts buffers, time off, closures, and existing bookings (see [[Availability Engine]]). |
| `Setup.gs` | One-time `setup()` that creates the data spreadsheet, builds tabs, seeds `Config`, and registers the first manager; also `addEmployee(...)`. |
| `Auth.gs` | Identity resolution and the three authorization guards. The security choke point. |
| `Code.gs` | Web-app entry point: `doGet` routing, the `include()` partial helper, and the `getServerInfo` probe. |
| `*Api.gs` (`AvailabilityApi`, `BookingApi`, `TimeOffApi`) | The client-callable feature APIs. Each entry point starts with an Auth guard, does all I/O through `SheetDAL`, converts time via `TimeUtil`, soft-deletes, and writes an audit entry. |
| `Notifications.gs`, `Reminders.gs`, `Triggers.gs` | Email notifications, reminder sweeps, and the time-driven trigger wiring (see [[Notifications]]). |
| `Reporting.gs` | Reporting/aggregation queries for the manager view. |
| `ui/*.html` | The web UI: page skeleton, styles, shared + per-role client JavaScript, and the two role views. |

## Conventions (follow these)

These conventions are load-bearing — breaking them tends to introduce security
holes, timezone bugs, or data corruption.

### 1. Server functions re-derive identity — never trust the client

The app is deployed **Execute as: Me (owner)** with **Anyone within domain**
access, so any rostered user could call any server function from devtools.
Therefore every client-callable function **re-derives who the caller is** from the
Google session and re-checks the role. Never accept an email/role/`userId` from
the client as proof of identity. The three guards in `Auth.gs`:

| Guard | Meaning |
|---|---|
| `requireAuthorized()` | Caller must be on the roster (any role), else throws. |
| `requireManager()` | Caller must be a manager, else throws. |
| `requireSelfOrManager(userId)` | Caller must be the manager **or** the specific employee whose data is being touched. |

Each returns the resolved user context (or throws). Start every client-callable
function with the appropriate guard. A `userId` argument is fine as a *target*
selector — but the guard, not the argument, decides whether the call is allowed.

### 2. All Sheet I/O goes through `SheetDAL`

Nothing else opens the spreadsheet directly. `SheetDAL` reads each tab in one
`getValues()` call, maps rows to objects via the **live header row** (so manual
column reordering in the Sheet doesn't corrupt reads), caches hot reads in
`CacheService`, and invalidates that cache on every write. Writes order columns
by `HEADERS[name]` and resolve rows by `PK[name]`.

The booking write-path runs inside `withScriptLock(fn)`, which serializes the
read–validate–write critical section and `flush()`es pending writes so the next
lock holder reads committed state. See [[Booking and Concurrency]].

### 3. All time goes through `TimeUtil`

Instants are stored as **UTC epoch-millisecond integers** in `*Ms` columns;
recurring times are wall-clock `"HH:mm"` strings in the single business timezone
(`Config.timeZone`, default `Europe/London`). The client never deals with
timezone math — the `*Api.gs` layer converts wall-clock input to epoch-ms (and
back) via `TimeUtil`, in the business tz. Intervals are **half-open**
`[startMs, endMs)`.

### 4. Soft-delete + audit

Rows are never physically deleted. Cancel/delete sets the row's `active` boolean
to `false`, preserving history. Every mutation also appends an `AuditLog` entry
(actor, action, entity, timestamp, before/after JSON).

### 5. HTML partials via `include()` and conditional role scripts

`Code.gs` exposes `include(name)`, which returns a file's content for use inside
`ui/Index.html` via `<?!= include('ui/Styles') ?>` etc. `doGet` injects the
chosen view file name into the `Index` template (`t.viewFile`), so `Index`
includes either the manager or employee partial — plus the matching per-role
JavaScript — based on the resolved role. Shared JS (`ui/JsCommon.html`) loads on
every page; role-specific JS (`ui/JsManager.html` / `ui/JsEmployee.html`) loads
only with its view.

## Routing, `google.script.run`, and shared client helpers

### `doGet` routing

`doGet(e)` in `Code.gs`:

1. Resolves the user context via `getCurrentUserContext()`. If that throws
   (roster unreadable — usually `setup()` hasn't run), it serves the
   **"Setup required"** notice (`ui/Notice`).
2. If the caller isn't on the roster (`!ctx.authorized`), it serves the
   **"No access"** notice.
3. Otherwise it picks `ui/Manager` for `ROLES.MANAGER`, else `ui/Employee`.
4. Builds a small `bootstrap` object (`state`, `email`, `role`, `userId`,
   `displayName`, `tz`, `companyName`, `notice`), serializes it to JSON, and
   injects it into the `Index` template — escaping every `<` in the JSON via
   `replace(/</g, '\\u003c')` so it can't break out of the page `<script>`.
5. Returns the evaluated template with the title set, a viewport meta tag, and
   `setXFrameOptionsMode(ALLOWALL)`. Combined with `<base target="_top">` in
   `Index`, this lets the app live inside the Google Sites iframe.

The bootstrap is read on the client as `window.FSW`. See [[Security and Permissions]] for the identity model behind step 1.

### `google.script.run` and `gcall`

The browser calls server functions through `google.script.run`. `JsCommon.html`
wraps that callback API in a promise helper:

```js
// gcall('serverFunctionName', arg1, arg2, ...) -> Promise
gcall('getServerInfo').then(function (info) { /* ... */ });
```

`gcall` slices off the method name, attaches `withSuccessHandler` / `withFailureHandler`,
and applies the remaining arguments. Use it for every server round-trip.

### Shared JsCommon helpers

`ui/JsCommon.html` loads on every page and provides:

| Helper | Purpose |
|---|---|
| `gcall(method, ...args)` | Promise wrapper around `google.script.run`. |
| `toast(msg, isError)` | Brief on-screen message (red when `isError`). |
| `initTabs()` | Scoped tab switching: a `.tab` toggles the `.panel` matching its `data-target` within its `.view`. |
| `initChrome()` | Populates the top bar (brand, name, role badge, notice) from `window.FSW`. |
| `checkConnection()` | Calls `getServerInfo` and shows the "✓ Connected" status line. |
| `$id(id)` / `$val(id)` | `getElementById` / its value. |
| `errMsg(e)` | Normalises an error to a message string. |
| `busy(btn, p)` | Disables a button while a promise runs; re-enables on settle. |
| `isoDate(d)` | Local `yyyy-MM-dd` for date inputs. |
| `affMsg(base, affected)` | Composes a save message warning about newly-flagged bookings. |
| `collectSpan(p)` | Reads a full-day/partial "span" form by id prefix `p` into `{scope, …}`. |
| `syncSpanFields(p)` | Shows/hides a span form's fields based on its scope select. |

The connection check is the fastest way to confirm identity resolution and the
client↔server channel are both healthy.

## Adding a new client-callable feature

The pattern is: **server function with a guard → call it from the UI with `gcall`.**

### 1. Server side (in the relevant `*Api.gs`)

```js
/**
 * Example: cancel an addition the caller owns.
 * @param {string} userId    target employee
 * @param {string} additionId
 * @return {Object}
 */
function cancelAddition(userId, additionId) {
  const ctx = requireSelfOrManager(userId);          // 1. guard: re-derive identity
  const row = findById(SHEETS.ADDITIONS, additionId); // 2. all I/O via SheetDAL
  if (!row || String(row.userId) !== String(userId)) throw new Error('Unknown addition.');
  updateById(SHEETS.ADDITIONS, additionId, { active: false }); // 3. soft-delete
  logAudit({ entityType: ENTITY.ADDITION, entityId: additionId, action: AUDIT_ACTION.DELETE,
             actorUserId: ctx.userId, actorEmail: ctx.email, atMs: Date.now(), before: row }); // 4. audit
  return { ok: true };
}
```

Checklist for any new entry point:

- Start with the right Auth guard; use the returned context, not client-supplied identity.
- Do every read/write through `SheetDAL` (`readObjects`, `findById`,
  `appendObject`, `updateById`); never touch `SpreadsheetApp` directly.
- Convert wall-clock input to epoch-ms via `TimeUtil`; store in `*Ms` columns.
- Mutate by soft-delete (`active: false`) and append an audit entry.
- For booking-style write paths that must not race, wrap the critical section in
  `withScriptLock(...)` (see [[Booking and Concurrency]]).

### 2. Client side (in `JsManager.html` / `JsEmployee.html`)

```js
busy(btn, gcall('cancelAddition', FSW.userId, additionId))
  .then(function () { toast('Removed.'); reload(); })
  .catch(function (e) { toast(errMsg(e), true); });
```

Then document the new function's signature in [[API Reference]] and add a test
in `Tests.gs` (see [[Testing]]).

## Documentation build

The per-phase docs in `docs/` are authored as Markdown and converted to Word.

- `tools/md2docx.py` — converts a single `.md` source to `.docx`.
- `tools/build-docs.sh` — rebuilds all the `.docx` files from their `.md`
  sources.

These tools live in `tools/` and are excluded from `clasp push` by
`.claspignore`, so they never reach the Apps Script project. Run
`tools/build-docs.sh` after editing any `docs/*.md` to refresh the generated
Word copies.
